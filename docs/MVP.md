# 《野排模拟器》MVP 设计文档

> 文档版本：0.4
> 项目阶段：最小可玩原型
> MVP 场景范围：仅两个核心场景——路线选择场景、战斗/事件场景。
> 关联文档：[游戏设计总览](GAME_DESIGN.md) · [角色数值与成长](CHARACTER_AND_PROGRESSION.md) · [场景与界面流程](SCENE_DESIGN.md) · [技术架构](SYSTEM_ARCHITECTURE.md)

## 1. MVP 一句话定义

玩家在随机生成的 Roguelike 路线中选择下一关，并在每个事件里通过指令卡影响三名不完全服从的 AI 队友，尽量带着物资和队伍活到撤离。

MVP 的核心不是完整局外系统，也不是复杂场景表现，而是验证两个问题：

1. 路线选择是否能形成“下一关去哪”的风险收益判断。
2. 战斗/事件中，玩家出牌后 AI 服从、犹豫或偏离是否有趣且可解释。

## 2. MVP 范围原则

本版本主动压缩场景数量，只保留局内最小闭环。

### 2.1 只做两个场景

| 场景 | 必须实现 | 主要职责 |
| --- | --- | --- |
| 路线选择场景 | 是 | 展示随机路线，从可达节点中选择下一关 |
| 战斗/事件场景 | 是 | 展示事件，选择指令卡，结算 AI 行为和结果 |

MVP 的实际循环是：

```text
路线选择
→ 选择一个可达节点
→ 战斗/事件
→ 选择指令卡或触发事件行为
→ 结算 AI 服从、偏离、事故和收益
→ 返回路线选择
→ 重复直到撤离或失败
```

### 2.2 暂不作为 MVP 场景

以下内容可以用临时数据、调试按钮、弹窗或后续版本实现，不作为 MVP 必须场景：

- 主菜单。
- 行动准备。
- 队友匹配展示。
- 卡牌查看场景。
- 角色查看场景。
- 完整对局结算场景。
- 局外大厅。
- 设置场景。

原型可以启动后直接进入路线选择场景，并使用默认角色、默认卡组和随机队友。

## 3. MVP 要验证的问题

MVP 只验证以下假设：

1. 玩家能理解路线图的分叉与汇合，并愿意根据威胁、收益和队伍状态选择下一关。
2. 玩家在战斗/事件场景中选择卡牌后，能感受到“我在指挥，但队友不是傀儡”。
3. AI 偏离指令时，玩家能看到 1—3 条主要原因，而不是觉得系统在乱判。
4. 战斗、搜索、关系和事故事件可以共用同一个战斗/事件场景完成。
5. 一局 5—8 个节点内能出现至少一次明显的队友偏离和一次路线风险取舍。

若玩家看不懂 AI 为什么违抗，优先改反馈；若玩家每次都能稳定控住队伍，优先提高事故热度和队友冲突。

## 4. 单局最小流程

MVP 不需要完整局外准备。开局时由测试配置生成一局。

```text
生成玩家角色、三名 AI 队友、初始 RunState
→ 生成随机路线图
→ 进入路线选择场景
→ 玩家选择下一节点
→ 进入战斗/事件场景
→ 展示事件情报
→ 玩家选择一张指令卡
→ AI 计算接受、犹豫或偏离
→ 结算事件结果
→ 更新队伍状态、物资和路线进度
→ 返回路线选择场景
→ 抵达撤离节点后成功，玩家死亡或撤离窗口耗尽后失败
```

## 5. 场景一：路线选择场景

路线选择场景是 MVP 的战略层。玩家只做一个主要决策：下一关去哪。

建议路径：`res://scenes/match/route_map.tscn`

### 5.1 场景职责

- 展示本局随机生成的路线图。
- 标记当前所在节点、已完成节点、可达节点和锁定节点。
- 展示每个可达节点的基础预览信息。
- 允许玩家选择一个可达节点进入战斗/事件场景。
- 展示全队关键状态，帮助玩家判断是否冒险。

路线选择场景不结算战斗、不出牌、不改变队友状态；它只负责“选择下一事件”。

### 5.2 路线结构

MVP 路线参考常见 Roguelike 分层路线图，从左到右推进。每层有 `2—4` 个节点，节点之间通过线连接，形成分叉和汇合。

```text
起点 → 第 1 层 → 第 2 层 → 第 3 层 → 第 4 层 → 撤离层
          2—4       2—4       2—4       2—4       1—2
```

玩家不是每层任选全部节点，而是只能选择当前节点连出去的下一层节点。

示例：

```text
                 [交火区] ─────→ [空投争夺]
               ↗          ↘       ↗
[起点] ─→ [废弃仓库]        [安全屋] ─→ [最终撤离]
      ╲        ↘          ↗       ↘
       ─→ [受伤路人] ─────→ [临时撤离点]
```

### 5.3 路线生成规则

MVP 使用 5 层路线：

1. 第 1—4 层每层生成 `2—4` 个事件节点。
2. 第 5 层生成 `1—2` 个撤离或高风险终点节点。
3. 起点连接第 1 层所有节点。
4. 普通节点连接下一层 `1—2` 个节点。
5. 少数分叉节点可连接 3 个节点。
6. 每个非终点节点至少有一条出边，避免死路。
7. 每个生成节点至少能从起点抵达。
8. 地图整体至少有 2 次有效分叉。
9. 允许多个节点汇合到同一节点。
10. 同一路径上不能连续出现 3 个相同类型节点。

如果生成结果不满足规则，使用同一种子的派生值重新生成。

### 5.4 节点类型

MVP 只保留 5 类节点，避免内容摊太开。

| 类型 | 作用 | 示例 |
| --- | --- | --- |
| 战斗 | 消耗状态，获得主要收益 | 遭遇敌方小队 |
| 搜索 | 获得物资，提高贪婪和暴露 | 搜索废弃仓库 |
| 关系 | 改变信任、怨气、凝聚力 | 稀有物资归属争执 |
| 事故 | 状态过差时放大矛盾 | 队友擅自追击 |
| 撤离 | 尝试结束本局 | 临时撤离点、最终撤离 |

补给、情报、精英等类型后续再拆分。MVP 可把它们作为事件标签挂在上述 5 类节点上。

### 5.5 节点预览信息

每个可达节点至少显示：

- 节点名称或模糊名称。
- 节点类型。
- 预计威胁：低 / 中 / 高。
- 预计收益：低 / 中 / 高。
- 特殊标签：如“高物资”“容易争执”“可能伏击”。
- 队友态度预测：想去、抗拒、无所谓、看不准。

示例：

```text
废弃仓库
类型：搜索
威胁：中
收益：高
标签：高物资 / 可能争抢

阿强：很想去（物资欲）
小刀：不太愿意（受伤）
老周：态度不明
```

队友态度只是预测，真正是否服从在战斗/事件场景结算。

### 5.6 路线选择场景布局

```text
┌──────────────────────────────────────────────────────────┐
│ 节点 2/5  威胁 35  凝聚力 56  物资 720  事故热度 22      │
├─────────────────────────────────────────┬────────────────┤
│                                         │ 节点详情       │
│       ○────○────○                       │ 类型：搜索     │
│      ╱ ╲    ╲  ╱                        │ 威胁：中       │
│ ●───○   ╲────◎────○                    │ 收益：高       │
│  ╲   ╲      ╱ ╲                         │ 队友态度       │
│   ○───○────○   ○                        │ [确认前往]     │
│                                         │                │
└─────────────────────────────────────────┴────────────────┘
```

MVP 不要求精美地图动画，只要节点、连线、可达状态和详情信息清晰。

## 6. 场景二：战斗/事件场景

战斗/事件场景是 MVP 的执行层。所有战斗、搜索、关系、事故和撤离节点都进入这个场景处理。

建议路径：`res://scenes/match/encounter.tscn`

### 6.1 场景职责

- 展示当前事件的文本、背景和已知情报。
- 展示玩家、三名 AI 队友和全队状态。
- 展示当前可用指令卡。
- 玩家选择一张指令卡或触发事件默认行为。
- 预览 AI 对该指令的大致态度。
- 结算 AI 服从、犹豫、偏离和事故。
- 展示结果原因与状态变化。
- 返回路线选择场景，或进入撤离/失败结果。

### 6.2 场景阶段

```text
事件进入
→ 情报展示
→ 玩家选择指令卡
→ AI 意向预览
→ 可选干预：劝说 / 施压 / 放任
→ 行动结算
→ 结果展示
→ 返回路线选择
```

MVP 可以先实现“选择指令后直接结算”，劝说和施压作为第二优先级。

### 6.3 指令卡

MVP 使用固定 8 张指令，不做抽卡、稀有度和构筑。

| 指令 | 主要效果 | 主要风险 |
| --- | --- | --- |
| 集火 | 提高战斗协同 | 高生存欲队友可能后撤 |
| 稳守 | 降低伤亡和威胁 | 收益较低 |
| 侧袭 | 提高进攻收益 | 暴露和事故热度提高 |
| 搜索 | 获得物资 | 提高贪婪压力和暴露 |
| 禁止舔包 | 压制物资行为 | 高物资欲队友增加怨气 |
| 跟紧我 | 提高凝聚力 | 降低搜索收益 |
| 救援 | 尝试救助倒地或受伤成员 | 全队承担额外风险 |
| 撤离 | 推进撤离或结束本局 | 高物资欲/表现欲队友可能拒绝 |

卡牌不是独立场景。MVP 在战斗/事件场景底部直接展示可用卡牌，点击卡牌弹出简短详情即可。

### 6.4 事件类型如何共用同一场景

| 节点类型 | 场景中的表现 | 主要结算 |
| --- | --- | --- |
| 战斗 | 敌人出现，选择战术指令 | 生命、压力、威胁、物资 |
| 搜索 | 区域可搜刮，选择搜刮或稳住队伍 | 物资、贪婪压力、暴露 |
| 关系 | 队友发生争执或分配冲突 | 信任、怨气、凝聚力 |
| 事故 | 某个已积累问题爆发 | 伤害、队伍分散、状态剧变 |
| 撤离 | 尝试带人和物资离开 | 成功、部分成功或失败 |

同一个 `encounter.tscn` 根据事件配置切换文本、背景、可用指令权重和结算规则。

### 6.5 战斗/事件场景布局

```text
┌──────────────────────────────────────────────────────────┐
│ 废弃仓库  威胁 35  凝聚力 56  物资 720  暴露 18          │
├──────────────┬──────────────────────────┬────────────────┤
│ 阿强         │                          │ 玩家：医生     │
│ HP/压力/信任 │      事件描述与结果       │ 体力/专注      │
│ 意向：想搜   │                          │ 职业能力       │
│              │      这里可能有高价值物资 │                │
│ 小刀         │                          │ 当前情报       │
│ 老周         │                          │                │
├──────────────┴──────────────────────────┴────────────────┤
│ [集火] [稳守] [搜索] [禁止舔包] [跟紧我] [救援] [撤离]  │
└──────────────────────────────────────────────────────────┘
```

### 6.6 结果反馈

每次结算后必须展示三层反馈：

1. 发生了什么。
2. 为什么发生。
3. 哪些状态变化了。

示例：

```text
小刀没有执行“稳守”，独自追击敌人。

主要原因：
+ 表现欲较高
+ 认为敌人已经残血
- 当前对你的信任不足

结果：
小刀生命 -18
小刀压力 +12
队伍凝聚力 -6
事故热度 +10
```

如果没有这类解释，AI 偏离就会显得像纯随机惩罚。

## 7. MVP 数据模型

MVP 使用完整角色文档的子集，不一次性实现所有属性。

### 7.1 玩家

```text
PlayerRunStats
  role_id
  health: 0—100
  stamina: 0—100
  focus: 0—100
  commandPressure: 0—100

  command: 0—100
  authority: 0—100
  empathy: 0—100
  insight: 0—100
  judgment: 0—100
  combat: 0—100
  medical: 0—100
```

MVP 可暂不实现 `mobility` 和 `logistics` 的完整效果。

### 7.2 AI 队友

```text
Teammate
  id
  name
  personality_id

  health: 0—100
  stress: 0—100
  panic: 0—100
  trust: 0—100
  resentment: 0—100
  morale: 0—100
```

MVP 至少提供 6 种人格模板：老六、莽夫、惜命怪、团队型、指挥瘾、赌徒。

### 7.3 全队状态

```text
RunState
```

## 8. AI 决策 MVP 公式

MVP 使用“指令接受分 + 执行稳定度”两段式。

### 8.1 指令接受分

```text
指令接受分 =
35
+ player.command × 0.22
+ player.authority × 0.12
+ teammate.trust × 0.20
+ run.cohesion × 0.14
+ teammate.discipline × 0.10
- teammate.stress × 0.16
- teammate.panic × 0.18
- teammate.resentment × 0.20
- 指令与个人驱动力冲突 × 0.25
- player.commandPressure × 0.08
+ 卡牌和事件修正
+ random(-10, 10)
```

结果：

| 分数 | 行为 |
| --- | --- |
| 60 以上 | 倾向服从 |
| 45—59 | 犹豫，可被劝说或施压 |
| 44 以下 | 偏离，选择个人效用最高行为 |

### 8.2 执行稳定度

```text
执行稳定度 =
行动基础值
+ 对应能力贡献
+ player.judgment × 0.10
+ run.cohesion × 0.10
+ teammate.morale × 0.08
- run.threat × 0.15
- run.exposure × 0.10
- 事件隐藏风险
+ random(-12, 12)
```

队友服从不代表必然成功。高威胁、高暴露或隐藏风险仍可能让行动失败。

## 9. 事故系统 MVP

MVP 至少实现 5 种事故：

| 事故 | 触发倾向 | 结果 |
| --- | --- | --- |
| 伏击 | 高暴露、搜索失败、战斗节点 | 受伤、威胁上升 |
| 争抢物资 | 高贪婪压力、高物资欲 | 怨气、信任下降、私藏 |
| 擅自追击 | 高表现欲、高士气 | 单人受伤或额外收益 |
| 恐慌撤退 | 高恐慌、高生存欲 | 凝聚力下降，可能提前撤离 |
| 甩锅争吵 | 连续失败、高怨气 | 队伍凝聚力下降 |

事故热度增加公式：

```text
事故热度增加 =
节点基础危险
+ threat / 10
+ exposure / 12
+ 平均压力 / 15
+ greedPressure / 18
- player.judgment / 20
```

事故不是每次都触发，但事故热度越高，事件越容易进入更坏分支。

## 10. MVP 角色

MVP 只提供两个玩家角色。若没有行动准备场景，可在配置中选择默认角色。

### 医生

```text
command 50
authority 35
empathy 70
insight 55
judgment 60
combat 35
medical 80
```

- 每局首次有队友倒地时，将其恢复至 30 生命。
- 使用救援或治疗类效果时，额外降低恐慌。
- 代价：战斗贡献低，施压能力弱。

### 突击手

```text
command 50
authority 65
empathy 35
insight 45
judgment 45
combat 80
medical 25
```

- 使用集火、侧袭等进攻指令时，战斗结算提高。
- 进攻成功时，提高凝聚力或士气。
- 代价：进攻失败时额外提高威胁和事故热度。

## 11. 胜利与失败

### 成功

- 玩家在撤离节点成功撤离。
- 带出的 `lootValue` 计为本局收益。
- 若有队友一起撤离，额外记录队伍收益。

### 失败

- 玩家生命归零。
- 路线推进到终点前队伍崩溃。
- 撤离事件失败且无后续可达节点。

MVP 可以用简单结果弹窗显示成功或失败，不需要独立结算场景。

## 12. MVP 内容范围

### 必须实现

- 2 个核心场景：
  - 路线选择场景。
  - 战斗/事件场景。
- 5 层随机路线。
- 每层 2—4 个节点，包含分叉和汇合。
- 5 类节点：战斗、搜索、关系、事故、撤离。
- 2 个玩家角色：医生、突击手。
- 3 名随机 AI 队友。
- 至少 6 种 AI 人格模板。
- 8 张固定指令卡。
- 至少 10 个事件配置。
- AI 服从、犹豫、偏离。
- 事故热度和至少 5 种事故。
- 结果原因展示。
- 成功撤离和失败弹窗。

### 暂不实现

- 主菜单、局外大厅、行动准备场景。
- 完整卡牌查看场景。
- 完整角色选择场景。
- 完整结算场景。
- 卡牌抽取、稀有度、构筑。
- 装备词条和背包格。
- 永久队友养成。
- PVP、联网匹配。
- 实时射击和复杂地图移动。
- 长剧情、语音和过场动画。

## 13. MVP 实施顺序

1. 固定测试数据：默认玩家、三名 AI、8 张指令卡。
2. 战斗/事件场景：先打通一个事件的展示、出牌、结算和结果反馈。
3. 路线选择场景：先用固定路线选择下一节点。
4. 两个场景互相切换：路线 → 事件 → 路线。
5. 接入随机路线生成。
6. 扩展到 10 个事件配置。
7. 接入 AI 人格和偏离原因展示。
8. 接入事故热度和事故分支。
9. 接入撤离节点与成功/失败弹窗。

先把两个场景的循环跑通，再增加内容。不要先做大厅、卡牌库或复杂局外成长。

## 14. 原型验收标准

- 启动后能直接进入路线选择场景。
- 路线图稳定生成 5 层，每层 2—4 个节点。
- 路线存在清晰分叉和汇合，不出现意外死路。
- 玩家能从可达节点中选择下一关。
- 选择节点后进入战斗/事件场景。
- 玩家能选择一张指令卡并完成结算。
- AI 至少会出现服从、犹豫、偏离三种结果。
- 每次偏离能展示至少 1 条主要原因。
- 事件结算能更新生命、压力、信任、怨气、凝聚力、威胁或物资。
- 战斗/事件结束后能返回路线选择场景。
- 抵达撤离节点后能显示成功。
- 玩家死亡或队伍崩溃后能显示失败。
- 单局能在 5—10 分钟内完成。

这一版 MVP 的目标很窄：只证明“路线选择 + 指令事件”这个骨架好玩。只要这个骨架成立，后续再补主菜单、卡牌查看、角色选择、结算和局外成长。
