# 场景与界面流程设计

> 文档版本：0.1  
> 目标：定义基础游戏场景、界面职责、场景流转和 MVP 实现边界。  
> 关联文档：[游戏设计总览](GAME_DESIGN.md) · [MVP](MVP.md) · [技术架构](SYSTEM_ARCHITECTURE.md)

> 同步状态：本文档仍保留旧版“行动准备/指令/服从”流程。当前核心方向已在 [游戏设计总览](GAME_DESIGN.md) 与 [MVP](MVP.md) 中改为“玩家是三人野排之一，事件场景围绕 AI 意图、玩家行动、简单喊话、互坑事故、背包归属和撤离姿态展开”。后续需要同步场景职责和 UI 布局。

## 1. 场景设计原则

- 一个全屏场景只负责一个主要决策。
- 局内状态由 `MatchController` 持有，切换界面时不能重新生成对局。
- 查看信息尽量使用弹窗或侧栏，不随意离开当前流程。
- 所有随机路线在开局时由种子生成，确保可以复现和调试。
- 决策前展示可能风险，决策后解释实际结果。
- MVP 优先保证信息清楚，不依赖复杂 3D 场景和角色动画。

## 2. 场景总览

| 场景 | 类型 | MVP | 主要职责 |
| --- | --- | --- | --- |
| 启动场景 | 流程场景 | 必须 | 初始化资源、存档和全局服务 |
| 主菜单/局外大厅 | 全屏场景 | 必须 | 开始行动、查看角色和卡牌 |
| 行动准备 | 全屏场景 | 必须 | 选择角色、指令组和难度 |
| 队友匹配 | 过渡场景 | 可简化 | 展示本局三名 AI 队友 |
| 关卡路线选择 | 全屏局内场景 | 必须 | 在随机路线中选择下一事件 |
| 主游戏 | 全屏局内场景 | 必须 | 处理事件、出牌、AI 响应和结算 |
| 卡牌查看/指令库 | 全屏局外场景 | 必须 | 查看已解锁指令和效果 |
| 对局结算 | 全屏场景 | 必须 | 展示撤离结果、收益与成长 |
| 解锁展示 | 弹窗/子场景 | 可简化 | 展示新角色、卡牌和专长 |
| 队伍详情 | 弹窗/侧栏 | 必须 | 查看 AI 属性、状态和关系 |
| 卡牌详情 | 弹窗 | 必须 | 查看单张卡牌完整说明 |
| 暂停与设置 | 覆盖层 | 必须 | 设置、返回菜单和退出确认 |
| 教程/图鉴 | 全屏局外场景 | 后续 | 回顾机制、人格和事件记录 |

## 3. 整体场景流程

```text
启动
  ↓
主菜单 / 局外大厅
  ├→ 卡牌查看
  ├→ 角色查看
  ├→ 设置
  └→ 行动准备
          ↓
      队友匹配
          ↓
      关卡路线选择
          ↓
        主游戏
          ↓
      关卡路线选择
          ↓
       重复 3—8 次
          ↓
    撤离成功 / 玩家失败
          ↓
       对局结算
          ↓
       解锁展示
          ↓
    主菜单 / 再来一局
```

局内只有“关卡路线选择”和“主游戏”反复切换。它们共享同一个 `RunState`，不能把路线和事件分别当成两局。

## 4. 启动场景

建议路径：`res://scenes/bootstrap/bootstrap.tscn`

### 职责

- 初始化 `ContentDB`、存档和设置。
- 校验角色、卡牌、人格和事件资源。
- 显示工作室标识与加载状态。
- 加载成功后进入局外大厅。

### 异常处理

- 首次启动：创建默认存档。
- 存档损坏：提示玩家并备份损坏文件。
- 内容配置错误：开发环境直接报告缺少的资源 ID。

启动场景不包含游戏逻辑，也不作为长期存在的根场景。

## 6. 行动准备场景

建议路径：`res://scenes/meta/loadout.tscn`

### 主要决策

1. 选择玩家角色。
2. 查看角色属性、被动和专属指令。
3. 选择指令组。
4. 选择行动难度或地图主题。
5. 确认开始。

### MVP 规则

- 可选角色：医生、突击手。
- 默认携带 8 张固定指令。
- MVP 可以只允许查看指令，不开放自由构筑。
- “开始行动”前必须通过配置完整性检查。

### 界面状态

```text
LoadoutSelection
  selected_role_id
  selected_command_ids[]
  selected_difficulty
  selected_mission_id
```

该状态在开始行动时转换为新的 `RunState`，未确认前不写入局内状态。

## 7. 队友匹配场景

建议路径：`res://scenes/match/squad_intro.tscn`

### 目的

在进入路线前，让玩家快速认识三名随机队友。这个场景承担“建立第一印象”的作用，不只是加载画面。

### 每名队友展示

- 昵称和头像。
- 能力倾向，如“擅长战斗”。
- 一条可见人格线索，如“看见好东西容易走不动路”。
- 初始信任状态。
- 一句体现性格的入队台词。

不直接展示全部精确驱动力。侦察员或高洞察角色可以看到更准确的信息。

### MVP 简化

三张队友卡依次翻开，玩家点击“出发”进入路线选择。不能在此更换队友。

## 8. 关卡路线选择场景

建议路径：`res://scenes/match/route_map.tscn`

这是局内的战略层。玩家在每个事件结算后返回此场景，从当前节点选择下一项。

### 8.1 路线结构

路线参考常见 Roguelike 的分层路线图，使用从左向右推进的有向无环图。每层包含 2—4 个事件节点，相邻层之间通过有限连线形成分叉和汇合。

```text
起点 → 第 1 层 → 第 2 层 → 第 3 层 → 第 4 层 → 最终撤离
          2—4       2—4       2—4       2—4
          个节点     个节点     个节点     个节点
```

MVP 建议生成 4—6 层，每层随机生成 2—4 个事件节点。玩家只能沿当前节点的连线前往下一层，因此一次可能有 1—3 个可达选项，而不是直接选择下一层的全部节点。

路线应同时包含：

- 分叉：一个节点连接到多个下一层节点。
- 汇合：多个上一层节点连接到同一个下一层节点。
- 独立路线段：部分节点只属于某一条风险或收益路线。
- 重新汇合点：允许不同路线在中后段回到共同节点。

玩家进入路线时可从第一层的 2—4 个起始节点中选择。完成节点后不能返回上一层。

示例：

```text
                 [交火区] ─────→ [空投争夺]
               ↗          ↘       ↗
[起点] ─→ [废弃仓库]        [安全屋] ─→ [最终撤离]
      ╲        ↘          ↗       ↘
       ─→ [受伤路人] ─────→ [临时撤离点]
```

在这个结构中，“废弃仓库”可以前往“交火区”或“安全屋”；“交火区”和“受伤路人”又都可能汇合到“安全屋”。路线选择的意义来自提前观察后续节点，而不是每次只比较眼前事件。

### 8.2 节点类型

| 类型 | 图标建议 | 主要内容 |
| --- | --- | --- |
| 战斗 | 准星 | 高威胁，战斗与物资收益 |
| 搜索 | 箱子 | 获得物资，提高贪婪压力 |
| 关系 | 对话气泡 | 信任、争执、队伍分配 |
| 补给 | 医疗箱 | 治疗、减压或交换资源 |
| 情报 | 雷达 | 揭示后续节点或降低风险 |
| 事故 | 警告标记 | 已有矛盾的高强度结算 |
| 精英 | 骷髅/红色准星 | 高风险高收益事件 |
| 撤离 | 出口 | 尝试结束本局 |
| 未知 | 问号 | 进入前只提供模糊线索 |

### 8.3 随机生成规则

推荐生成流程：

1. 根据行动难度确定层数，MVP 使用 5 层。
2. 每层生成 `2—4` 个节点，第一层建议固定为 2 个。
3. 起点连接第一层全部节点，让玩家先选择路线方向。
4. 普通节点默认连接下一层 `1—2` 个节点，少数关键分叉可连接 3 个。
5. 每个非终点节点至少有一条出边，避免死路。
6. 下一层每个节点至少有一条入边，避免不可达节点。
7. 允许多个节点连接到同一节点，形成路线汇合。
8. 优先连接位置相邻的节点，减少难以阅读的长距离交叉线。
9. 每张地图至少包含 2 次真正的分叉选择，不能全程只有单一路线。
10. 同一路径不能连续出现超过 2 个相同类型。
11. 前两层不生成强制事故或最终撤离。
12. 第 3 层开始允许生成临时撤离点。
13. 最后一层至少包含一个最终撤离节点。
14. 每条从起点到终点的完整路线至少包含一个收益节点和一个风险节点。
15. 生成后执行可达性、死路和分叉数量校验；失败则使用同一种子派生值重新生成。

伪代码：

```text
generate_route(seed, depth = 5):
  layers = []

  for depth_index in 0..<depth:
    node_count = random_range(2, 4)
    layer = generate_weighted_nodes(depth_index, node_count)
    layers.append(layer)

  connect_start_to_first_layer(layers[0])
  connect_adjacent_layers(
    layers,
    normal_outgoing = 1..2,
    branch_outgoing = 3
  )
  enforce_generation_rules(layers)

  if not validate_reachability(layers)
     or has_dead_end(layers)
     or count_meaningful_branches(layers) < 2:
    regenerate_with_derived_seed()

  return layers
```

### 8.4 事件权重

节点不是完全均匀随机。推荐初始权重：

| 类型 | 前期 | 中期 | 后期 |
| --- | ---: | ---: | ---: |
| 战斗 | 25 | 30 | 35 |
| 搜索 | 35 | 25 | 15 |
| 关系 | 20 | 20 | 15 |
| 补给 | 10 | 10 | 10 |
| 情报 | 10 | 10 | 5 |
| 事故 | 0 | 5 | 10 |
| 撤离 | 0 | 条件生成 | 必定至少 1 个 |

还应根据当前状态动态修正：

- 平均生命较低时，提高补给节点出现概率。
- 凝聚力较低时，提高关系和事故节点概率。
- 已持有高价值物资时，提高撤离节点概率，但同时提高伏击事件概率。
- 连续两层没有战斗时，下一层战斗权重上升。

动态修正只能影响尚未揭示的节点，不能暗中替换玩家已经看到的路线。

### 8.5 信息展示

节点卡需要展示：

- 事件名称或模糊名称。
- 类型图标。
- 预计威胁：低/中/高。
- 预计收益：低/中/高。
- 特殊标签，如“适合侦察”“队友感兴趣”。
- 通往该节点的连接。
- 当前是否可达。

玩家选中节点后，右侧详情栏展示队友态度：

```text
废弃仓库
预计威胁：中
预计收益：高

阿强：很想去（物资倾向）
小刀：不太愿意（当前受伤）
老周：态度不明

[确认前往]
```

队友态度是预测，不是最终服从结果。选择路线代表“前往该区域”，进入主游戏场景后才选择具体指令。

### 8.6 路线地图状态

```text
RouteState
  seed
  layers[]
  current_node_id
  visited_node_ids[]
  revealed_node_ids[]
  available_node_ids[]

RouteNode
  id
  layer_index
  event_id
  type
  threat_preview
  reward_preview
  outgoing_node_ids[]
  incoming_node_ids[]
  state
```

节点状态包括：

- `LOCKED`
- `AVAILABLE`
- `SELECTED`
- `COMPLETED`
- `SKIPPED`
- `HIDDEN`

路线只负责选择事件，不直接执行事件效果。

进入路线时，第一层的 2—4 个节点为 `AVAILABLE`。完成某个节点后，只有它的 `outgoing_node_ids` 对应节点变为 `AVAILABLE`，其余同层节点保持锁定或变为 `SKIPPED`。

正常情况下可达节点可以只有 1 个；这表示玩家此前的路线选择把队伍带入了一段固定路线。地图整体仍必须提供至少两次分叉，避免整局路线失去策略性。

### 8.7 推荐布局

```text
┌──────────────────────────────────────────────────────────┐
│ 节点 3/5  威胁 42  凝聚力 58  物资 1,250   [队伍详情]   │
├─────────────────────────────────────────┬────────────────┤
│                                         │ 节点详情       │
│       ○────○────○                       │ 威胁：中       │
│      ╱ ╲    ╲  ╱                        │ 收益：高       │
│ ●───○   ╲────◎────○                    │ 队友态度       │
│  ╲   ╲      ╱ ╲                         │                │
│   ○───○────○   ○                        │ [确认前往]     │
│                                         │                │
├─────────────────────────────────────────┴────────────────┤
│ 路线图例 / 当前撤离窗口 / 特殊状态                       │

```

## 9. 主游戏场景

建议路径：`res://scenes/match/encounter.tscn`

主游戏场景负责单个事件从展示到结算的完整过程。

### 9.1 场景阶段

```text
事件进入
→ 情报展示
→ 玩家选择指令
→ AI 意向预览
→ 可选干预
→ 行动结算
→ 结果展示
→ 返回路线 / 撤离结算 / 失败结算
```

### 9.2 推荐布局

```text
┌──────────────────────────────────────────────────────────┐
│ 节点名  威胁  凝聚力  物资  撤离状态       [暂停]        │
├──────────────┬──────────────────────────┬────────────────┤
│ 队友 1       │                          │ 玩家角色       │
│ 生命/压力    │      事件画面与文本       │ 职业能力       │
│ 信任/意向    │                          │ 临时效果       │
│              │      行动结果演出         │                │
│ 队友 2       │                          │ 事件情报       │
│ 队友 3       │                          │                │
├──────────────┴──────────────────────────┴────────────────┤
│ [指令卡] [指令卡] [指令卡] [指令卡]   [查看全部]        │
└──────────────────────────────────────────────────────────┘
```

### 9.3 交互规则

- 玩家点击卡牌后先进入预览状态，再次确认才提交。
- 选中卡牌时，高亮目标、预计影响和可能冲突的队友。
- AI 意向阶段显示“愿意/犹豫/拒绝”的预测，不提前展示最终随机结果。
- 若允许劝说、强制或职业能力，只能在意向阶段使用。
- 结算期间锁定输入。
- 结算完成后显示主要状态变化和 AI 行为原因。

### 9.4 结果反馈

结果分三层展示：

1. 发生了什么：攻击成功、队员受伤、获得物资。
2. 为什么发生：队友服从或偏离的主要因素。
3. 改变了什么：生命、压力、信任、凝聚力和威胁变化。

示例：

```text
小刀没有执行“稳守”，独自追击敌人。

主要原因：
+ 认为敌人已经残血
+ 表现欲较高
- 当前对你的信任不足

结果：
小刀生命 -18
小刀压力 +12
队伍凝聚力 -6
```

### 9.5 事件表现

MVP 使用以下组合即可：

- 一张事件背景图。
- 角色头像和状态动画。
- 文本、图标、数字跳字。
- 简单镜头震动和音效。

不要求每个事件制作独立可操作 2D/3D 地图。

## 10. 卡牌查看场景

建议路径：`res://scenes/meta/command_library.tscn`

### 职责

- 查看所有已解锁和未解锁指令。
- 按类型、角色、标签筛选。
- 查看卡牌完整规则和升级/专长变化。
- 从行动准备场景进入时，可支持选择或替换指令。

### 卡牌分类

- 战术
- 行动
- 关系
- 情报
- 角色专属

### 推荐布局

```text
┌──────────────────────────────────────────────────────────┐
│ 指令库       [搜索] [类型筛选] [角色筛选]       已解锁 8 │
├──────────────────────────────────────┬───────────────────┤
│ [卡牌] [卡牌] [卡牌] [卡牌]         │ 卡牌大图          │
│ [卡牌] [卡牌] [锁定] [锁定]         │ 名称 / 类型       │
│ [卡牌] [卡牌] [卡牌] [卡牌]         │ 效果 / 代价       │
│                                      │ AI 冲突提示       │
│                                      │ 解锁条件          │
├──────────────────────────────────────┴───────────────────┤
│ [返回]                              [装备/替换，仅准备时] │
└──────────────────────────────────────────────────────────┘
```

### 卡牌信息规范

每张卡至少显示：

- 名称和类型。
- 一句话用途。
- 目标规则。
- 主要效果。
- 使用代价、次数或冷却。
- 容易发生冲突的 AI 倾向。
- 来源角色和解锁条件。

避免只写“效果提高 15%”而不说明提高什么、持续多久、在哪个结算阶段生效。

## 11. 队伍详情弹窗

建议路径：`res://scenes/ui/squad_panel.tscn`

可从路线选择和主游戏场景打开。

### 展示内容

- 玩家与三名队友。
- 生命、压力、信任和个人物资。
- 已识别的人格特征。
- 临时效果与伤势。
- 最近一次服从或偏离原因。

打开弹窗时暂停界面计时，但不修改局内状态。高洞察只增加可展示的信息，不创建第二套队友数据。

## 12. 对局结算场景

建议路径：`res://scenes/results/match_results.tscn`

### 结算顺序

1. 显示撤离成功或行动失败。
2. 展示每名成员结局。
3. 统计带出物资和损失。
4. 展示账号经验与角色熟练度。
5. 展示关键事件时间线。
6. 触发解锁弹窗。

### 关键事件时间线

建议保留 3—5 条最有代表性的记录：

- 首次违抗。
- 最大收益。
- 队员倒地或救援。
- 事故触发。
- 最终撤离决定。

### 操作

- 再来一局：保留上次角色和指令配置，重新生成种子。
- 返回大厅。
- 查看详细行动记录。

## 13. 暂停与设置覆盖层

建议路径：`res://scenes/ui/pause_menu.tscn`

### MVP 功能

- 继续游戏。
- 音量设置。
- 显示设置。
- 操作说明。
- 放弃本局。
- 返回主菜单。

放弃本局必须二次确认，并明确说明未撤离物资会损失。设置直接写入局外配置，不能写入 `RunState`。

## 14. 后续可扩展场景

### 角色与专长场景

查看职业详情、熟练度和互斥专长。MVP 可合并进行动准备场景。

### 行动记录场景

查看历史对局、随机种子、使用角色、撤离收益和关键事故。

### 人格图鉴

记录已经遇到的 AI 人格和典型行为。首次遭遇前只显示模糊轮廓，避免直接泄露全部规则。

### 教程场景

使用三个固定短事件分别教授路线选择、指令与 AI 偏离、撤离。教程使用固定种子和固定队友，不依赖普通随机生成。

### 每日行动场景

所有玩家使用同一随机种子、角色限制和特殊规则进行挑战。只在核心单机循环稳定后考虑。

## 15. 场景与弹窗的职责边界

| 功能 | 实现方式 | 原因 |
| --- | --- | --- |
| 选择下一事件 | 全屏路线场景 | 这是局内主要战略决策 |
| 执行单个事件 | 全屏主游戏场景 | 需要完整注意力和结算演出 |
| 查看全部卡牌 | 全屏卡牌场景 | 信息量大，需要筛选 |
| 查看单张卡牌 | 弹窗 | 不应打断当前流程 |
| 查看队友详情 | 侧栏/弹窗 | 保留当前事件上下文 |
| 设置与暂停 | 覆盖层 | 返回后恢复原界面 |
| 显示新解锁 | 弹窗 | 依附于结算流程 |

## 17. 场景切换数据

场景之间只传递轻量参数或读取统一状态：

```text
SceneRequest
  scene_id
  transition
  payload
```

示例：

```text
路线 → 主游戏
payload:
  selected_route_node_id

主游戏 → 路线
payload:
  completed_event_id

主游戏 → 结算
payload:
  result_type
```

完整队友、路线和物资状态仍保存在 `RunState`，不通过字典在场景间反复复制。

## 18. MVP 实施顺序

1. 主菜单。
2. 行动准备。
3. 队友匹配简版。
4. 关卡路线选择及固定测试路线。
5. 主游戏场景及一个完整事件。
6. 随机路线生成。
7. 队伍详情和卡牌详情弹窗。
8. 卡牌查看场景。
9. 对局结算。
10. 暂停与设置。

先用固定路线打通完整流程，再接入随机生成。这样路线生成问题不会掩盖事件和状态流转问题。

## 19. 场景验收标准

- 玩家能从主菜单完整完成一局并返回大厅。
- 每层路线稳定生成 2—4 个节点，并形成可读的分叉与汇合。
- 所有非终点节点至少有一条出边，所有生成节点均可从起点抵达最终撤离。
- 地图整体至少提供两次有效分叉，沿线不存在意外死路。
- 已展示的路线不会因状态变化偷偷替换。
- 路线选择与事件指令是两个清晰、不同的决策。
- 主游戏结算后能解释 AI 的主要行为原因。
- 打开卡牌或队伍详情不会丢失当前局内状态。
- 撤离成功与失败都能正确进入结算并保存局外奖励。
- 所有主要场景支持键鼠操作，基础按钮具有焦点与返回行为。
